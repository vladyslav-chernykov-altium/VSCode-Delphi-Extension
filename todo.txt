================================================================================
  rsm2pdb - project todo
================================================================================

Goal: convert Delphi .rsm (Remote Symbol) debug-info files into DWARF (v1)
and PDB (v2) so VSCode + gdb (and later cppvsdbg / WinDbg / x64dbg) can
debug Delphi binaries.

Scope locked for v1:
  - Modern Delphi (XE / 10.x / 11+)
  - x64 only
  - Symbols + line numbers + basic types (primitives, records, enums,
    pointers, arrays)
  - DWARF output, injected as new sections into a copy of the EXE
  - VSCode debugging via ms-vscode.cpptools cppdbg + mingw-w64 gdb

Out of scope for v1 (deferred):
  - 32-bit (x86)
  - Sets, variants, dynamic arrays, interfaces, managed strings as
    native structs
  - PDB output
  - In-place EXE editing
  - CFI / unwind tables
  - Inlined-subroutine DIEs

[x] DWARF emit: extend CU range to cover function symbols (2026-05-26).
         Commit d6d6abf.

         Symptom: on Altium AdvPCB, breakpoints in the .dpr's
         `begin..end` block (= the compiler-generated unit-init
         function AdvPCB.AdvPCB) hit correctly but VSCode silently
         failed to navigate to the source. gdb's `info source`
         returned "No current source file." Comparable third-party
         tool map2pdb worked on the same .map, pointing at DWARF
         emission as the gap.

         Root cause: src/dwarf/dwarf_emitter.cpp cuRange() derived
         [low_pc, high_pc) only from line-entry addresses. Delphi's
         compiler-generated unit-init function often has sparse / no
         line entries -- so its code range fell outside any CU's
         [low_pc, high_pc). gdb never consulted that CU's line
         program for those addresses, and returned no source.

         Fix: also absorb every function-symbol address attributed to
         the CU into its low_pc/high_pc. Now any address inside a
         function known to be part of this unit triggers the line-
         program lookup -- which surfaces the file:line mapping
         whenever line entries exist nearby. Cosmetic gap remains for
         bare `begin`/`end` lines (Delphi doesn't emit per-line
         entries for them); deferred behind "synthesise line entries
         at function boundaries" follow-up.

         Verified on AdvPCB.dll: breakpoint in
            library AdvPCB;
            begin
              WriteLn(...);   <-- breakpoint here, source displays
            end.
         now shows source. Tests: 27 / 2021, all green.

[x] Real-project scale: 120 MB DLLs (Altium AdvPCB) (2026-05-26).
         Commits d9fbe67 (perf), 088a11d (proc-scanner robustness),
         d6d6abf (CU range fix).

         Smoke-tested the whole pipeline on the Altium PCB engine:
            DLL   120 MB     .map   267 MB     .rsm  543 MB
            946,961 publics   38,062 line tables   3,937 unique units

         Before fixes: process killed at 4 minutes with no output.
         After fixes:  full parse + populate + decorate + DWARF emit
                      + PE inject in 52 seconds.

         Phase breakdown (post-fix):
            23.1s   parse .map
             1.2s   populate model       (was 80s with O(N*M) walk)
            26.8s   parse .rsm
             0.0s   decorateTypes        (was infinite without lookup index)
             0.6s   DWARF emit           (info=31MB, line=9MB)
             0.7s   PE inject            (158 MB output DLL)

         Algorithmic fixes:
           1. rsm::Reader gained unordered_map<VA, idx> for variables
              and procedures. findVariableAt / findProcedureAt go
              from O(N) to O(1). Without this, decorateTypes on
              1.9M publics x 478k variables would have been ~9e11 ops.
           2. map::populate bucketed publics by LONGEST matching
              unit-name prefix, eliminating ~57x duplication; then
              merged line tables per module into ONE CompileUnit
              instead of one-CU-per-segment.
           3. Variable-scan dump filter widened from 8 MB to 1 GB so
              big DLLs' user-VA-range procedures are visible.
           4. Procedure-record sub-scanner validates Pascal-identifier
              names, skipping records like compiler-generated hidden
              `Self` parameters whose name bytes contain NUL (these
              would corrupt our DWARF byte stream via DW_FORM_string
              truncation -- the 6 abbrev-misread CUs that gdb
              reported are all fixed).

         Verified end-to-end: gdb resolves Pascal-qualified symbols
         (System.SysUtils.Abort, JclAnsiStrings.CharIsLower) and
         maps to source:line cleanly. Output DLL loadable, breakpoint
         resolves.

[x] M2 phase A: function parameters + locals (2026-05-22).
         Commit dfe98ca.

         Pipeline now decodes RSM procedure records (tag 0x28):
           name + VA + hash + return-type-marker
           param sub-records (tag 0x21)
           local sub-records (tag 0x20)
           end marker (0x63)

         Each param/local carries (name, type_marker, stack_offset).
         The stack offset is encoded in 2-byte stride units relative
         to a frame anchor at rbp+16. Conversion:
            real_byte_offset = (rsm_offset_signed / 2) + 16
         Discovered by cross-checking RSM offsets against asm
         spill addresses in Geometry.Add (preferred VAs 0x20/0x30
         in RSM correspond to asm [rbp+0x20]/[rbp+0x28]).

         DWARF emission:
           DW_TAG_subprogram WITH children + DW_AT_frame_base = rbp+16
           DW_TAG_formal_parameter children with DW_OP_fbreg location
           DW_TAG_variable children (typed locals) with same shape

[x] M2 phase B-lite: typed globals via primitives only (2026-05-22).
         Goal: Delphi globals like `S: Integer` show as Integer in
         VSCode (instead of `int`), and records/enums show as
         `byte[N]` arrays so users at least see the underlying bytes
         in the right width.

           - RSM header parser (8 fields + EXE-path string).
           - RSM primitive-table decoder (28 built-in Delphi types,
             stable IDs across builds).

             (5-byte plain primitive, 12-byte extended, 6-byte
             non-primitive). Captures per-unit type_marker and
             trailer_type_id. VA decoded as stored_u32 >> 4.
           - decorateTypes() adapter: groups primitives by per-unit
             marker, computes MIN(next-symbol-gap) per group, maps
             1/2/4/8 -> Byte/Word/Integer/Int64, falls back to
             byte[N] for non-standard sizes and all non-primitives.
           - DWARF emitter: 4 new abbrevs (typed_variable, base_type,
             array_type, subrange_type). Type DIEs emitted before
             vars; DW_AT_type via DW_FORM_ref_addr.
           - rsm2pdb dwarf auto-discovers sibling .rsm and decorates
             if present.

                       same-size sibling -- documented D-017 trade-off.

         D-017 limits accepted (deliberate scope cut):
           - Cardinal mistyped as Integer (same size, sign merged).
           - UInt64/Double mistyped as Int64.
           - Single (4-byte float) padded to 8, mistyped as Int64.
           - 1-byte types (Byte/ShortInt/Boolean) padded to 2 by
             alignment, displayed as Word.

         marker -> primitive-table-id bridge (per-unit type table
         not yet decoded) or RTTI parsing from the EXE -- both
         deferred as separate milestones.

----- Methodology -----

Two parsers feed the same model::Module:
  src/map/  - Delphi .map text parser (lines, public symbols, segments)
  src/rsm/  - Delphi .rsm binary parser (types, locals, params,
              additional symbols, source paths)
The .map and .rsm are produced from the same compiler IR so they
agree by construction. We use .map for the parts that are awkward
to extract from RSM (line tables - see D-016).

================================================================================
  KEY DECISIONS LOG
================================================================================

D-001  DWARF first, PDB later (smaller surface area; lets us reach a
       working VSCode debug session faster).
D-002  C++20 + MSVC + LLVM 18.1.8.
D-003  x64 only for v1.
D-004  Fidelity v1 = symbols + lines + basic types (records, enums).
D-005  PE-section injection is the planned delivery mechanism for DWARF.
       To be confirmed by step 6 spike.
D-006  Intermediate model (src/model/) is debugger-agnostic; both DWARF
       and PDB backends read the same Module.
D-007  Use VS-bundled CMake + Ninja (no separate install).
D-008  Build LLVM from source against MSVC (RTTI + EH on) - prebuilt
       LLVM binaries are clang-built and ABI-incompatible with MSVC
       in subtle ways.
D-009  DWARF language code = DW_LANG_Pascal83 for now; revisit if gdb
       behavior is surprising.
D-010  LLVM build profile = "lean": drop BUILD_TOOLS / BUILD_UTILS /
       dylibs / optional features (libxml2, zstd, zlib, libedit,
       libpfm, terminfo, bindings, ocamldoc). Keep RTTI + EH on.
       Target ~15-25 min build, ~1 GB installed instead of 30-60 min
       and ~3 GB. We only use Support / Object / BinaryFormat / MC /
       DebugInfoDWARF as static libs.
D-011  Considered dropping LLVM entirely for v1 (DWARF-only). Decision:
       keep LLVM. Trade-off accepted: 2-3 days coding saved vs. ~1 GB
       disk + ABI sensitivity. PDB backend (deferred) genuinely needs
       LLVM's DebugInfo/PDB, so dropping then re-adding is wasted
       work.
D-012  Architecture confirmed by spikes 6a/6b/6c. The five DWARF
       sections .debug_info / _abbrev / _line / _str / _line_str are
       sufficient for source-level debugging via gdb on Windows.
       Long section names will use the COFF string-table "/N"
       convention. Sections marked DISCARDABLE | READ |
       CNT_INITIALIZED_DATA.
D-013  Delphi fixture build via Delphi MSBuild (NOT direct dcc64).
       examples/01_hello/hello.dproj + build.cmd that sources
       rsvars.bat. Enables further per-example CI automation.
D-014  Modern Delphi RSM format does NOT use TDS "FB09" magic;
       it uses ASCII "CSH7". JCL is therefore primarily an
       intellectual-history reference, not a direct translation
       source. The parser will be largely reverse-engineered.
D-015  Two complementary RSM-format docs:
         - rsm-format.txt at project root: running lab notebook,
           append-only, raw hex dumps + hypotheses + [FACT]/[GUESS]
           tags. Lives next to todo.txt.
         - docs/02-rsm-format-notes.md: polished design view of
           what we believe the format IS, kept clean.
D-017  M2 phase B-lite: typed globals via PRIMITIVES ONLY.
       Don't try to decode RSM records / enums / classes in this
       milestone. Globals whose RSM type_id resolves to a primitive
       (Integer, Byte, Double, ...) get a proper DW_TAG_base_type.
       Everything else (records, enums) falls back to a byte[N] array
       sized by next-symbol gap. Rationale:
         - delivers visible value fast (Integer globals show real
           values, not "int" coincidence);
         - validates the RSM-decoding approach with the simplest
           record class;
         - leaves the harder record/enum decoding for a focused later
           milestone (step 11) where we replace the byte[N] fallbacks.
       Order: P1-lite (header + primitive types) -> P2-lite (bind
       globals to type IDs) -> D-globals-typed (DWARF wire-up).

D-016  Dual input: .map AND .rsm.
       Empirically: line-address tuples from the .map are NOT stored
       in the .rsm as raw uint32s. The line table uses a non-trivial
       encoding (likely LEB128 deltas or state-machine like DWARF).
       Reversing it would cost 3-5 days for zero fidelity gain - the
       .map is authoritative linker output, generated from the same
       compiler IR. Use:
         .map -> line tables, public symbols, segments
         .rsm -> types, locals, params, source paths, anything else
       Both populate the same model::Module; rsm2pdb at runtime
       expects hello.exe + hello.map + hello.rsm sitting together.
