================================================================================
  rsm2pdb - project todo
================================================================================

Goal: convert Delphi .rsm (Remote Symbol) debug-info files into DWARF (v1)
and PDB (v2) so VSCode + gdb (and later cppvsdbg / WinDbg / x64dbg) can
debug Delphi binaries.

Scope locked for v1:
  - Modern Delphi (XE / 10.x / 11+)
  - x64 only
  - Symbols + line numbers + basic types (primitives, records, enums,
    pointers, arrays)
  - DWARF output, injected as new sections into a copy of the EXE
  - VSCode debugging via ms-vscode.cpptools cppdbg + mingw-w64 gdb

Out of scope for v1 (deferred):
  - 32-bit (x86)
  - Sets, variants, dynamic arrays, interfaces, managed strings as
    native structs
  - PDB output
  - In-place EXE editing
  - CFI / unwind tables
  - Inlined-subroutine DIEs

[x] M3 -- PDB backend (LLVM-backed). Default backend in the VSCode
         extension; DWARF is now opt-in via rsm2pdb.backend = dwarf.

         Pipeline (rsm2pdb pdb <map> <exe-in-place> <pdb>):
            1. Parse PE for section headers (also become the PDB's
               SectionMap + DbgStream[SectionHdr]).
            2. Parse .map -> publics + line tables + module_segments.
            3. Optional .rsm -> proc records (params + locals).
            4. Compose Modules (one per Pascal unit), each with
               functions, locals, line entries. Convert .map
               (segment_id, segment_offset) -> (PE section index,
               section-relative offset) via image_base + RVA math.
            5. Resolve source paths (Delphi writes basenames in .map)
               against --src-search dirs and mapDir / mapDir/.. /
               mapDir/../.. fallbacks. fs::canonical() makes them
               absolute -- cppvsdbg won't bind BPs to relative paths.
            6. writePdb (src/pdb/pdb_writer.cpp): LLVM PDBFileBuilder
               + InfoStreamBuilder + DbiStreamBuilder + TpiStreamBuilder
               + IpiStreamBuilder + GSIStreamBuilder. Emits:
                  - S_PUB32 (publics, qualified names, function-flag set)
                  - S_GDATA32 globals (UNQUALIFIED names; Int32 type)
                  - Per-module S_GPROC32 + S_FRAMEPROC + S_REGREL32 +
                    S_END (Int32 type, RBP frame ptr)
                  - C13 line subsections (DebugChecksums MD5 +
                    DebugLines per function)
                  - SectionContribs sorted by (ISect, Off) before submit
                  - DbgStream[SectionHdr] = COFF section headers verbatim
            7. injectPdbReference (src/pe/pe_pdb_injector.cpp): adds a
               new .debug section containing IMAGE_DEBUG_DIRECTORY +
               RSDS payload (magic "RSDS" + GUID + age + PDB basename)
               and patches OptionalHeader.DataDirectory[DEBUG].

         VSCode extension (vscode-ext): new rsm2pdb.backend setting
         (default "pdb"). buildRunner branches on it; extension.ts
         composes a cppvsdbg launch config for PDB mode (no MIMode/
         setupCommands -- VS native engine reads PDB directly). The
         --src-search dirs come from the .dproj's unitSearchPaths +
         includeSearchPaths + dirname(dccReferences).

         Verified end-to-end on examples/02_two_units: breakpoints in
         .dpr and .pas hit, step-into enters Pascal source, Locals
         shows A/B/Tmp/Result, Watch on `S` / `D` / `P1` resolves
         global vars (typed as Int32 raw view for now).

         Tests: test/unit/test_pdb_writer.cpp +
         test/unit/test_pe_pdb_injector.cpp. 31/31 pass.

         Pitfalls encountered & documented in CLAUDE.md gotchas 19-21

[x] RSM proc parser: real-world scale; ~125k methods now carry args+locals
         (2026-05-26).

         Symptom: on Altium AdvPCB, breakpoints inside Delphi methods

         Diagnostic harness added (new subcommands):
            rsm2pdb diff-procs    -- VA-cross-check rsm procs vs .map
                                     functions; reports missing-in-rsm
                                     with raw-name presence in .rsm bytes.
            rsm2pdb probe-procs   -- hex + decoded byte dump for a
                                     stratified sample of missing procs.
            rsm2pdb analyze-procs -- single-pass histograms over the whole
                                     file (proc subtag distribution,
                                     trailer-length distribution, sub-
                                     record subtag distribution) for
                                     deriving application rules without
                                     full format RE.

         What the histograms showed:
            map functions:        301,829
            rsm proc candidates:  134,186  (VA-confirmed; upper bound)
            our parser pre-fix:     6,473  (2.1% of map -- 24x undercount)

         Four independent over-strict checks were each cutting deeply:

         1. Fixed `headEnd + 16` jump. The bytes between VA and the
            first sub-record have VARIABLE length (3..30 bytes
            empirically; only 16.4% of procs happen to be 5).
            Fix: scan-forward up to 96 bytes for a valid sub-record
            start OR a real 0x63 end marker (one whose next byte is a
            recognizable record tag 0x28/0x20/0x2A/0xFF/0x9C; bare
            0x63 bytes inside the trailer are common false positives).

         2. Proc-record sub-tag was hard-restricted to 0xA0. Histogram
            shows 0xE0 (1.5%, RTL allocators) and 0x80 (0.4%) are also
            valid. Fix: accept {0xA0, 0xE0, 0x80}.

         3. Sub-record subtag was hard-restricted to {0x66, 0x62}.
            Histogram shows 7 common values; 0x66 only 56% of the
            time. The strong invariant is the two zero bytes after
            the subtag, not the subtag value. Fix: accept any byte,
            require the two-zero invariant.

         4. Real Delphi RSMs contain additional sub-record tags
            (0x23, 0x25) for return-value metadata and enum-entry
            definitions after the param/local list. These don't fit
            our fixed 5-byte sub-record format. Fix: stop the walk
            gracefully on unknown tags (keep collected params/locals)
            instead of discarding the whole proc.

         Plus minor fixes:
           - Pascal-identifier set widened to include `@` (so
             System.@Abs, System.@DllMain etc. parse).
           - Between sub-records: scan-forward 0..6 bytes for next
             valid sub-record header to tolerate `var`/`out`/`const`
             parameter qualifier prefix bytes.
           - isInsideProcedure() replaced with std::upper_bound
             binary search (linear was O(N) per query, catastrophic
             at 134k procs in the variable-scan loop).

         PE injector fix:
           - The injector now RENAMES any pre-existing .debug_*
             sections to ".old_dw" before appending fresh ones.
             Without this, repeated re-injection produced a PE with
             two .debug_info sections; gdb read the FIRST one
             (stale) and reported "no locals or arguments" even
             when the new DWARF carried them correctly. The user-
             reported symptom was actually this masking bug, not
             the parser bug alone.

         Numbers post-fix:
            rsm procs:        159,462  (24.6x improvement)
            matched (VA):     125,732  (41.7% of map)
            walk failures:          0
            pipeline time:       ~39s  (vs ~52s pre-fix)

         The remaining 58% of map functions have no rsm proc record
         at all (compiler-generated stubs, RTTI helpers, virtual

[x] DWARF emit: extend CU range to cover function symbols (2026-05-26).
         Commit d6d6abf.

         Symptom: on Altium AdvPCB, breakpoints in the .dpr's
         `begin..end` block (= the compiler-generated unit-init
         function AdvPCB.AdvPCB) hit correctly but VSCode silently
         failed to navigate to the source. gdb's `info source`
         returned "No current source file." Comparable third-party
         tool map2pdb worked on the same .map, pointing at DWARF
         emission as the gap.

         Root cause: src/dwarf/dwarf_emitter.cpp cuRange() derived
         [low_pc, high_pc) only from line-entry addresses. Delphi's
         compiler-generated unit-init function often has sparse / no
         line entries -- so its code range fell outside any CU's
         [low_pc, high_pc). gdb never consulted that CU's line
         program for those addresses, and returned no source.

         Fix: also absorb every function-symbol address attributed to
         the CU into its low_pc/high_pc. Now any address inside a
         function known to be part of this unit triggers the line-
         program lookup -- which surfaces the file:line mapping
         whenever line entries exist nearby. Cosmetic gap remains for
         bare `begin`/`end` lines (Delphi doesn't emit per-line
         entries for them); deferred behind "synthesise line entries
         at function boundaries" follow-up.

         Verified on AdvPCB.dll: breakpoint in
            library AdvPCB;
            begin
              WriteLn(...);   <-- breakpoint here, source displays
            end.
         now shows source. Tests: 27 / 2021, all green.

[x] M2 phase A: function parameters + locals (2026-05-22).
         Commit dfe98ca.

         Pipeline now decodes RSM procedure records (tag 0x28):
           name + VA + hash + return-type-marker
           param sub-records (tag 0x21)
           local sub-records (tag 0x20)
           end marker (0x63)

         Each param/local carries (name, type_marker, stack_offset).
         The stack offset is encoded in 2-byte stride units relative
         to a frame anchor at rbp+16. Conversion:
            real_byte_offset = (rsm_offset_signed / 2) + 16
         Discovered by cross-checking RSM offsets against asm
         spill addresses in Geometry.Add (preferred VAs 0x20/0x30
         in RSM correspond to asm [rbp+0x20]/[rbp+0x28]).

         DWARF emission:
           DW_TAG_subprogram WITH children + DW_AT_frame_base = rbp+16
           DW_TAG_formal_parameter children with DW_OP_fbreg location
           DW_TAG_variable children (typed locals) with same shape

[x] M2 phase B-lite: typed globals via primitives only (2026-05-22).
         Goal: Delphi globals like `S: Integer` show as Integer in
         VSCode (instead of `int`), and records/enums show as
         `byte[N]` arrays so users at least see the underlying bytes
         in the right width.

         Pipeline added end-to-end:
           - RSM header parser (8 fields + EXE-path string).
           - RSM primitive-table decoder (28 built-in Delphi types,
             stable IDs across builds).
           - RSM variable-record scanner: three sub-forms
             (5-byte plain primitive, 12-byte extended, 6-byte
             non-primitive). Captures per-unit type_marker and
             trailer_type_id. VA decoded as stored_u32 >> 4.
           - decorateTypes() adapter: groups primitives by per-unit
             marker, computes MIN(next-symbol-gap) per group, maps
             1/2/4/8 -> Byte/Word/Integer/Int64, falls back to
             byte[N] for non-standard sizes and all non-primitives.
           - DWARF emitter: 4 new abbrevs (typed_variable, base_type,
             array_type, subrange_type). Type DIEs emitted before
             vars; DW_AT_type via DW_FORM_ref_addr.
           - rsm2pdb dwarf auto-discovers sibling .rsm and decorates
             if present.

                       Size-merged primitives (Cardinal/Single/Double/
                       UInt64/Boolean) display under their best-fit
                       same-size sibling -- documented D-017 trade-off.

         D-017 limits accepted (deliberate scope cut):
           - Cardinal mistyped as Integer (same size, sign merged).
           - UInt64/Double mistyped as Int64.
           - Single (4-byte float) padded to 8, mistyped as Int64.
           - 1-byte types (Byte/ShortInt/Boolean) padded to 2 by
             alignment, displayed as Word.
           - Char defaults to Integer when it's the last symbol in
             a unit (no next-gap to measure).
         Bytes shown are always correct; only the type-name display
         is best-effort. To remove these limits we'd need a precise
         marker -> primitive-table-id bridge (per-unit type table
         not yet decoded) or RTTI parsing from the EXE -- both
         deferred as separate milestones.

----- Methodology -----

Two parsers feed the same model::Module:
  src/map/  - Delphi .map text parser (lines, public symbols, segments)
  src/rsm/  - Delphi .rsm binary parser (types, locals, params,
              additional symbols, source paths)
The .map and .rsm are produced from the same compiler IR so they
agree by construction. We use .map for the parts that are awkward
to extract from RSM (line tables - see D-016).

[x] 21   DONE 2026-05-28. Interface-adjuster-thunk naming +
         .natstepfilter generation. Implemented exactly as
         described below.

         Context: when stepping into an interface method, Delphi
         routes the call through a tiny adjuster thunk
             add rcx, -ofs
             jmp TClass.Method
         that has no source mapping in RSM. Our VSCode extension
         walks through it via `skipUnknownSourceMode = "hybrid"`
         (2 stepIns) and lands the user in the method body. VS2022
         IDE doesn't load our extension, so the user there sees
         "Source Not Available" + disassembly until they press F11
         a few more times manually.

         Approach: pure rsm2pdb-side fix that also benefits cdb /
         cppvsdbg by giving the thunk a meaningful name.

           a) Thunk detection in the PE.
              Heuristic: a function with exactly two instructions
              matching the pattern
                  48 83 C1 NN          ; add rcx, imm8 (sign-ext)
                  E9 NN NN NN NN       ; jmp rel32
              (5 + 4 bytes total; sometimes 4 + 5 with REX flip).
              Walk all proc bytes between the lowest .map function
              VA and the highest, look for those byte patterns.
              Cross-reference the jmp target against the .map: if
              it lands on a real class-method, this is an adjuster
              thunk for THAT method.

           b) Synthesise a S_GPROC32 + S_PUB32 entry for the thunk
              in the PDB. Suggested naming:
                  <Class>.<Method>$Adjust_<Hex>
              e.g. iface.TMyClass.MyMethod$Adjust_FFFFFFE0
              Public name surfaces in Call Stack instead of the
              current `iface...TMyClass+0xa`.

           c) Generate <pdb-base>.natstepfilter alongside the PDB:
                  <?xml version="1.0" encoding="utf-8"?>
                  <StepFilter ...>
                    <Function>
                      <Name>.*\$Adjust_[0-9A-F]+$</Name>
                      <Action>NoStepInto</Action>
                    </Function>
                  </StepFilter>
              VS2022 reads this from the directory next to the
              PDB. Step Into through the thunk then tail-call-skips
              to the real method, exactly like our hybrid does.

           d) cppvsdbg picks up .natstepfilter the same way -- the
              VSCode hybrid-mode auto-skip becomes redundant for
              this specific case, but kept as fallback for RTL
              functions that natstepfilter can't easily express.

         Effort: ~half a day. Touches src/pdb/, src/cli/cli_cmd_pdb,
         needs no compose-side changes.

         Bonus: cleaner Call Stack -- adjuster thunks named
         meaningfully, not as offsets.

  KEY DECISIONS LOG
================================================================================

D-001  DWARF first, PDB later (smaller surface area; lets us reach a
       working VSCode debug session faster).
D-002  C++20 + MSVC + LLVM 18.1.8.
D-003  x64 only for v1.
D-004  Fidelity v1 = symbols + lines + basic types (records, enums).
D-005  PE-section injection is the planned delivery mechanism for DWARF.
       To be confirmed by step 6 spike.
D-006  Intermediate model (src/model/) is debugger-agnostic; both DWARF
       and PDB backends read the same Module.
D-007  Use VS-bundled CMake + Ninja (no separate install).
D-008  Build LLVM from source against MSVC (RTTI + EH on) - prebuilt
       LLVM binaries are clang-built and ABI-incompatible with MSVC
       in subtle ways.
D-009  DWARF language code = DW_LANG_Pascal83 for now; revisit if gdb
       behavior is surprising.
D-010  LLVM build profile = "lean": drop BUILD_TOOLS / BUILD_UTILS /
       dylibs / optional features (libxml2, zstd, zlib, libedit,
       libpfm, terminfo, bindings, ocamldoc). Keep RTTI + EH on.
       Target ~15-25 min build, ~1 GB installed instead of 30-60 min
       and ~3 GB. We only use Support / Object / BinaryFormat / MC /
       DebugInfoDWARF as static libs.
D-011  Considered dropping LLVM entirely for v1 (DWARF-only). Decision:
       keep LLVM. Trade-off accepted: 2-3 days coding saved vs. ~1 GB
       disk + ABI sensitivity. PDB backend (deferred) genuinely needs
       LLVM's DebugInfo/PDB, so dropping then re-adding is wasted
       work.
D-012  Architecture confirmed by spikes 6a/6b/6c. The five DWARF
       sections .debug_info / _abbrev / _line / _str / _line_str are
       sufficient for source-level debugging via gdb on Windows.
       Long section names will use the COFF string-table "/N"
       convention. Sections marked DISCARDABLE | READ |
       CNT_INITIALIZED_DATA.
D-013  Delphi fixture build via Delphi MSBuild (NOT direct dcc64).
       examples/01_hello/hello.dproj + build.cmd that sources
       rsvars.bat. Enables further per-example CI automation.
D-014  Modern Delphi RSM format does NOT use TDS "FB09" magic;
       it uses ASCII "CSH7". JCL is therefore primarily an
       intellectual-history reference, not a direct translation
       source. The parser will be largely reverse-engineered.
D-015  Two complementary RSM-format docs:
         - rsm-format.txt at project root: running lab notebook,
           append-only, raw hex dumps + hypotheses + [FACT]/[GUESS]
           tags. Lives next to todo.txt.
         - docs/02-rsm-format-notes.md: polished design view of
           what we believe the format IS, kept clean.
D-017  M2 phase B-lite: typed globals via PRIMITIVES ONLY.
       Don't try to decode RSM records / enums / classes in this
       milestone. Globals whose RSM type_id resolves to a primitive
       (Integer, Byte, Double, ...) get a proper DW_TAG_base_type.
       Everything else (records, enums) falls back to a byte[N] array
       sized by next-symbol gap. Rationale:
         - delivers visible value fast (Integer globals show real
           values, not "int" coincidence);
         - validates the RSM-decoding approach with the simplest
           record class;
         - leaves the harder record/enum decoding for a focused later
           milestone (step 11) where we replace the byte[N] fallbacks.
       Order: P1-lite (header + primitive types) -> P2-lite (bind
       globals to type IDs) -> D-globals-typed (DWARF wire-up).

D-016  Dual input: .map AND .rsm.
       Empirically: line-address tuples from the .map are NOT stored
       in the .rsm as raw uint32s. The line table uses a non-trivial
       encoding (likely LEB128 deltas or state-machine like DWARF).
       Reversing it would cost 3-5 days for zero fidelity gain - the
       .map is authoritative linker output, generated from the same
       compiler IR. Use:
         .map -> line tables, public symbols, segments
         .rsm -> types, locals, params, source paths, anything else
       Both populate the same model::Module; rsm2pdb at runtime
       expects hello.exe + hello.map + hello.rsm sitting together.
