  WORKING ASSUMPTIONS (revisit as we learn more)

- Target: modern Delphi (10.2+ Tokyo), Win64 (x86_64) only for v1.
- Endianness: LITTLE-ENDIAN throughout (TD32 was LE; assume same here).
- Address width: 64-bit for Win64 RSM. To verify.
- File is laid out as: fixed header + directory + subsections + ...
  but we haven't validated this; could be more like a stream of typed

================================================================================
  2026-05-22  Initial inspection of hello.rsm
================================================================================

Raw hex of first 64 bytes:

offset  hex                                              ascii
------  -----------------------------------------------  ----------------
0x0000  43 53 48 37 20 04 00 00 0e 00 00 00 01 00 00 00  CSH7 ...........
0x0010  17 52 b6 5c 4d 23 00 20 00 00 40 00 00 00 00 00  .R.\M#. ..@.....
0x0020  2e 5c 57 69 6e 36 34 5c 44 65 62 75 67 5c 68 65  .\Win64\Debug\he
0x0030  6c 6c 6f 2e 65 78 65 00 00 00 00 00 00 00 00 00  llo.exe.........

[FACT]   Magic is ASCII "CSH7" at offset 0. Probably stands for
         "Code Symbol Hierarchy v7" or similar. The "7" likely a
         format version.

[FACT]   This is NOT the legacy Borland TD32/TDS layout (which used
         "FB09" / "FB0A" magic and is what JCL's JclTD32.pas parses).
         Embarcadero replaced or substantially redesigned the format
         at some point. JCL is a hint but not a translation source.

[GUESS]  offset 0x04: uint32 0x00000420 = 1056. Probably an offset
         into the file (header end? directory start?). Plausible
         since 1056 is well past our 64-byte sampled header.

[GUESS]  offset 0x08: uint32 0x0000000E = 14. Maybe a count
         (subsections / modules / something).

[GUESS]  offset 0x0C: uint32 0x00000001 = 1. Maybe a sub-version,
         flag, or "module count" if 0x08 was section count.

[GUESS]  offset 0x10: uint32 0x5CB65217 = 1555471895 = 2019-04-17
         13:51:35 UTC. Looks like a Unix timestamp. Could be the
         RSM format epoch (when this version of the format was
         frozen) OR the build timestamp of this specific .rsm.

[QUESTION] offset 0x14: uint32 0x2000234D. No clean interpretation.
         If we read it as two uint16: 0x234D, 0x2000. 0x2000 might
         be alignment or flags. 0x234D unclear.

[GUESS]  offset 0x18: uint32 0x00400000. This is the canonical
         Win32 default PE ImageBase. Weird because our EXE is Win64
         with ImageBase 0x140000000. Maybe a legacy field; maybe
         a different VA the symbols are relative to; maybe a
         relocation hint.

[FACT]   offset 0x20: null-terminated ASCII path to the EXE this
         RSM was generated for. ".\Win64\Debug\hello.exe" here.
         Notice the relative path - the linker writes it as
         relative to the .dproj directory.

[QUESTION] Where does the path field end (max length)? Padded to
         some boundary? Need to check by varying the .dpr path.

[QUESTION] Right after the path, what comes? Zeros visible in the
         sample, but is that padding to alignment, or actual data?

         is all zeros (padding/reserved).

[FACT]   At offset 0x426 a structured block begins. The bytes

         follow at 0x42D.

[FACT]   The library/import search path list appears verbatim seven

           c:\dev\tools\embarcadero\studio\19.0\lib\Win64\release\EN\

           C:\Dev\Tools\Embarcadero\Studio\19.0\include\
         Each path is null-terminated. Path lists separated by another
         small structure (a few bytes between them).

[GUESS]  The uint16 0x0007 at offset 0x426 = count of path-list copies.
         If this stays at 7 for our simple project, it's a per-fixed
         count (six standard categories + "user/include" paths)
         indexed by some module-builder concept. Or it's the same list
         repeated because Delphi serializes each search path per
         purpose: unit search / namespace / hpp / include / object /
         resource. To verify by varying compile options.

[FACT]   Immediately after the path lists (around 0xADE): the
         DCC_UnitAliases string is stored as one big delimited list:
           DbiErrs=BDE;DbiProcs=BDE;DbiTypes=BDE;WinProcs=Winapi.Windows;
           WinTypes=Winapi.Windows;Generics.Defaults=System.Generics.Defaults;
           Generics.Collections=System.Generics.Collections

[FACT]   At offset 0xB90: the relative EXE path appears AGAIN
         (.\Win64\Debug\hello.exe). So this string is stored at least
         twice in the file (once near offset 0x20, once near 0xB90).

[FACT]   Most of the 3.5 MB is Delphi RTL: unit names (System,
         SysUtils, Winapi.Windows, ...) and imported symbols (
         @DelayLoadHelper2, @GetTls, kernel32.dll, VirtualAlloc, ...)
         starting around 0xBBF.

[FACT]   User code symbols live at the very END of the file
         (offsets 0x354F00+). Strings we see for hello:
           $unwind$_ZN5Hello3AddEii         at 0x354F11
           $pdata$_ZN5Hello3AddEii          at 0x354F34
           Finalization                     at 0x354F56
           hello                            at 0x354F79
           $unwind$_ZN5Hello12FinalizationEv at 0x354F90
           $pdata$_ZN5Hello12FinalizationEv at 0x354FBC
           $unwind$_ZN5Hello14initializationEv at 0x354FE7
           $pdata$_ZN5Hello14initializationEv  at 0x355015
           TPoint                           at 0x3550D4
           TColor                           at 0x35511F
           clGreen                          at 0x35513D
           clBlue                           at 0x355145
         Then several unit-name strings, finally section names
         (.itext .rdata .pdata .xdata) and source dir.

[FACT]   Delphi x64 uses ITANIUM-STYLE NAME MANGLING (not the
         legacy Borland mangling). Example:
            _ZN5Hello3AddEii    =  Hello::Add(int, int)
            _ZN5Hello14initializationEv = Hello::initialization()
         Important implication: when we emit DWARF, the DW_AT_name
         attribute should contain the DEMANGLED Pascal name
         ("Hello.Add" or just "Add"), not the mangled form. We need
         a small Itanium demangler (or use llvm::itaniumDemangle
         from the LLVM Demangle lib we already linked).

[FACT]   Win64 EH frames generate auxiliary symbols prefixed
         "$pdata$" (RUNTIME_FUNCTION entries) and "$unwind$"
         (UNWIND_INFO blobs). These are NOT user code; they're
         the unwinder's metadata. We should likely skip emitting
         DIEs for them (they have no source location to map to).

[FACT]   Records near offset 0x354F00..0x355100 use a small
         tag-length-string pattern:
            27 18 "$unwind$_ZN5Hello3AddEii("    (0x18 = 24 chars)
            27 17 "$pdata$_ZN5Hello3AddEii("     (0x17 = 23 chars)
            28 0C "Finalization"                  (0x0C = 12 chars)
            28 05 "hello"                         (0x05 = 5 chars)
         The TAG byte (0x27, 0x28, ...) likely identifies a symbol
         KIND. LENGTH is exactly 1 byte (so symbols are limited to
         255 chars). Names are NOT null-terminated within this
         record kind.

         Around the records there are more bytes between them - these
         must be the symbol's address / size / type-id / attributes.

[FACT]   At offset 0x355100+ we see a TColor enum record. Pattern:
            03 06 "TColor"
            <some bytes>
            05 "clRed" 07 "clGreen" 06 "clBlue"
         The leading 0x03 might be the enum-type tag, 0x06 = name
         length. The enumerator list uses raw length-prefix strings
         (5/"clRed" 7/"clGreen" 6/"clBlue") - clean Pascal-shortstring
         style.

[FACT]   The byte sequence right after "hello" looks like ACTUAL X64
         MACHINE CODE for Add():
            55 48 83 EC 10  48 8B EC  89 4D 20  89 55 28
            8B 45 20  03 45 28  89 45 08  ...  5D C3
         Decodes to:
            push rbp; sub rsp, 0x10; mov rbp, rsp;
            mov [rbp+0x20], ecx; mov [rbp+0x28], edx;
            mov eax, [rbp+0x20]; add eax, [rbp+0x28];
            mov [rbp+0x08], eax; ... pop rbp; ret
         This is Add()'s body. Why is it stored in the RSM?
         Possible reasons:
            a) Embedded for "edit & continue" / fix-up purposes.
            b) Stored as part of a symbol's value for compile-time
               constant inlining.
            c) Coincidence - bytes happen to look like code.
         To investigate.

[FACT]   Address 0x00025900 (hello.Add per .map) appears in the file
         right before the TColor record. Pattern:
            02 01  59 02 00  02  00 00 00 08 ...
         where 02 01 = uint16 0x0102 might be a "segment+kind" code
         and 01 59 02 00 = uint32 0x00025901 might be Add's address.
         Off by 1 from the .map's 0x25900 - could be (a) intentional
         (skipping prologue) or (b) my byte-alignment guess is off
         by 1.

            line 34  0x00025940     "40 59 02 00"  -> 0 hits
            line 35  0x0002595F     "5f 59 02 00"  -> 0 hits
            line 36  0x00025969     "69 59 02 00"  -> 0 hits
            line 37  0x00025973     "73 59 02 00"  -> 0 hits
            line 38  0x0002597A     "7a 59 02 00"  -> 1 false-positive hit
            line 39  0x00025991     "91 59 02 00"  -> 0 hits
            line 40  0x000259DE     "de 59 02 00"  -> 0 hits

[FACT]   The 1 hit for "7a 59 02 00" at file offset 0x1B9AC4 is a
         FALSE POSITIVE. Bytes around it show "ciexyzX/Y/Z" struct
         field names - it's the CIEXYZ color triple from
         Winapi.Windows GDI definitions. Address coincidence.

[GUESS]  The .rsm line-table encoding is non-trivial. Most likely
         candidates:
            - LEB128 deltas like DWARF's line state machine
            - run-length encoded (line_delta + addr_delta) byte pairs
            - per-source-file header + offset list relative to a
              function base
            - state machine with opcodes (DW_LNS_*-like)
         Reversing this would cost 3-5 days of careful work for
         zero fidelity gain - the .map is authoritative linker output
         covering exactly this.

[D-016 RECORDED]  Decision: use .map for line numbers + public
         symbols, .rsm for types + locals + params + source paths.
         Both feed model::Module. The .rsm parser does NOT need to
         decode line tables. Note this saves real time AND moves the
         RSM RE focus to the parts we genuinely need it for.

----- User-global symbol records (located at ~0x355b00) -----

[FACT]   User globals (S, D, C, P1, P2) are encoded near the end of the
         .rsm with a record shape:

            0x20 <namelen u8> <name bytes> 0x66 0x00 0x00 <payload>

         Concrete bytes from two_units.rsm:

            20 02 50 31 66 00 00 4d 06 87 75 3a 04           (P1)
            20 02 50 32 66 00 00 4d 06 07 76 3a 04           (P2)
            20 01 43    66 00 00 51 06 87 76 3a 04           (C)
            20 01 44    66 00 00 02 c7 76 3a 04 9c 09 99 1b f5 03 ff  (D)
            20 01 53    66 00 00 02 07 77 3a 04 9c 09 9d 1b f5 03 ff  (S)

         Cross-referenced with two_units.map .bss VAs (segment 0003,
         base 0x430000):
            P1 0003:A758  P2 0003:A760
            C  0003:A768  D  0003:A76C  S  0003:A770

[FACT]   The leading tag is 0x20 (record-kind = VARIABLE).
         The name-length byte is 1-byte unsigned (P1 has len=2; S has
         len=1).

[FACT]   The trailing `66 00 00` byte triple is constant across all
         five records. Probably a fixed delimiter or sub-tag (compare
         the 0x27/0x28 tags seen earlier near function-like records).

[GUESS]  Payload shape correlates with TYPE KIND, not symbol identity:
             P1, P2 (TPoint record)         -> 6-byte payload
             C      (TColor enum)            -> 6-byte payload
             D, S   (Integer primitive)      -> 12-byte payload
         The shorter form likely encodes (type_id_u16, VA_or_offset)
         only; the longer form may carry an inline type descriptor
         or extra primitive metadata.

[GUESS]  Inside the 6-byte form, bytes split as <u16_LE> + <u32_LE>:
             P1: typeId=0x064D  payload32=0x043A7587
             P2: typeId=0x064D  payload32=0x043A7607
             C : typeId=0x0651  payload32=0x043A7687
         P1 and P2 share typeId (both TPoint) - strong signal.
         BUT the payload32 doesn't match the .map VAs (P1 expected
         0x0043A758). The payloads ARE 8 bytes apart between P1 and P2
         which matches the .map spacing, just with an offset/encoding
         transform we haven't decoded yet (the high nibble pattern
         suggests possibly nibble-shifted or BCD-style encoding).
         Will revisit when we cross-reference more globals.

[QUESTION] What does the 12-byte primitive-typed payload encode? S and D
         share the first 5 bytes' SHAPE (02 ?? ?? 3a 04) and the trailer
         pattern (9c 09 ?? 1b f5 03 ff). The differing nibble in the

================================================================================
  2026-05-22 (cont)  Step 3: primitive type table located + decoded
================================================================================

Method: grep both fixtures for primitive names (Integer, Byte, Double, ...)
near the metadata stream. They cluster between offset 0x147F and 0x19BB in
hello.rsm. Each entry follows the record shape:

    0x2A <namelen u8> <name bytes> 0xA8 0x00 0x00 <hash u32?> <attrs> ...
        0x9C 0x13 <type_id_lo u8> <type_id_hi u8> <cpp_name_block> 0xFF

[FACT]   Primitive-record kind tag is 0x2A. Records are terminated by
         0xFF — BUT 0xFF also occurs inside record payloads (e.g.
         Cardinal's body contains `f6 ff 81 02`). The robust way to
         find the true terminator is to look forward for the NEXT
         record's start signature (0x2A <printable_namelen> <ASCII
         letter>) and treat the byte just before that as the
         terminator. The src/rsm/rsm_reader.cpp scanner does this.

[FACT]   The 3-byte sub-tag immediately after the name varies between
         primitives:
            0xA8 0x00 0x00  - most primitives (Boolean, Integer, Byte,
                              Word, Cardinal, Pointer, UInt64, ...)
            0x80 0x00 0x00  - WideChar, LongInt, LongWord
            0x88 0x00 0x00  - ShortInt, SmallInt, Single, Real48,
                              Extended80, Double, Real, Comp, Currency
         Pattern unclear. Possibly a "linkage flavour" or "cpp-mapping
         category". Not needed for our purposes.

[FACT]   The 0x9C 0x13 marker delimits the type-id field. The 2 bytes
         immediately following it are a u16-LE that uniquely identifies
         the primitive type ACROSS BUILDS — verified: same values appear
         in hello.rsm and two_units.rsm.

[FACT]   Type-id values observed in our Delphi 10.2 Tokyo fixtures
         (full primitive table, 28 entries):
            Boolean    0x08E4    AnsiChar   0x08E6    Char       0x08E8
            WideChar   0x08EA    ShortInt   0x08EC    SmallInt   0x08EE
            Integer    0x08F0    Byte       0x08F2    Word       0x08F4
            Cardinal   0x08F6    Pointer    0x08F8    LongInt    0x08FA
            LongWord   0x08FC    Int64      0x08FE    UInt64     0x0201
            NativeInt  0x0205    NativeUInt 0x0209    Single     0x020D
            Real48     0x0211    Extended80 0x0215    Extended   0x0219
            Double     0x021D    Real       0x0221    Comp       0x0225
            Currency   0x0229    ByteBool   0x024D    WordBool   0x0251
            LongBool   0x0255

[FACT]   Type-ids increment by 2 in the contiguous block 0x08E4..0x08FE.
         Then a 16-bit wrap or kind-switch causes the jump to 0x0201,
         after which increments are typically +4 (with gaps to 0x024D
         for the BoolN family). The exact ID-encoding rule is not yet
         understood but the values are stable per primitive — that's
         what we need for cross-referencing with variable records.

[FACT]   ByteBool / WordBool / LongBool sit in a SEPARATE 0x2A-record
         cluster after a gap (~0x195F in hello.rsm). They share the
         same 0xA8 sub-tag flavour and the 0x9C 0x13 marker convention.

[FACT]   Byte SIZE and SIGNEDNESS are NOT independently stored in the
         record (at least not in a place we've identified). We map
         name -> (size, kind) via a hardcoded table in
         src/rsm/rsm_reader.cpp — Delphi's primitive set is closed and
         well-known.

[QUESTION] Variable records carry a `9C 09 ?? ??` reference, not
         `9C 13 ?? ??`. Two distinct Integer-typed variables (S, D)
         also differ in the bytes after `9C 09` — so this 4-byte
         construct cannot be a plain type-id reference. To investigate
         in step 4 (variable -> type binding).

[QUESTION] What does the 4-byte field immediately after `0xA8 0x00 0x00`
         encode? Each primitive has a unique-looking 4-byte value
         (Boolean: 38 81 bc 62; Integer: a0 ce 73 b5; Byte: ea 26 ee 04).
         Plausibly a hash or RTTI signature. Not needed for B-lite.


================================================================================
  2026-05-22 (cont)  Step 4a: variable-record iterator + VA decoding
================================================================================

Method: scan whole metadata stream for variable-record signature
    0x20 <namelen 1..64> <printable ASCII name> 0x66 0x00 0x00 <payload>
Decode the payload according to its first byte.

[FACT]   Variable-record byte layout (verified against P1 / P2 / C / D / S
         in two_units.rsm and P / C / S in hello.rsm):

           0x20 <namelen u8> <pascal_name> 0x66 0x00 0x00 <payload>

         The payload form is selected by its first byte:

           if payload[0] == 0x02:                  -- PRIMITIVE form
              0x02 <shifted_va u32> <7 trailing bytes ending in 0xFF>
              total payload size = 12 bytes

           otherwise:                              -- NON-PRIMITIVE form
              <type_id u16-LE> <shifted_va u32>
              total payload size = 6 bytes

         There is no record-terminator byte; the record ends after the
         fixed-size payload and the next record's 0x20 tag follows
         immediately.

[FACT]   VA encoding for global variables: stored as a little-endian
         uint32 left-shifted by 4 bits relative to the actual
         virtual address. To decode:  actual_va = stored_u32 >> 4.

         Verified against all five user globals in two_units.rsm and
         all three in hello.rsm against the .map .bss addresses.

[FACT]   For non-primitive globals (records, enums, classes) the
         inline type_id is the same u16 you find in the primitive
         table's `9C 13 ?? ??` marker for primitives — i.e. a
         direct index into the RSM type table. Verified:
            P1 and P2 (both TPoint)  -> same type_id   0x064D
            C  (TColor enum)         -> different     0x0651
            FLanguages, LanguageArray -> their own RTL type_ids
         Hello has different per-unit type_ids for its TPoint/TColor
         (0x1A71 / 0x1A6D) because the types are declared inline in
         the .dpr unit rather than imported from System.Types.

[FACT]   For primitive-typed globals (Integer, Byte, ...) the
         variable record carries NO recognisable primitive-table
         type-id. The trailing `9C 09 <u16>` differs even between
         two Integer-typed variables (S = 0x1B9D, D = 0x1B99 in
         two_units), so this field is NOT a type reference -- likely
         a source-location hash or per-variable instance id.
         The primitive -> variable binding has to come from somewhere
         else; investigating in step 4b.

[FACT]   Same scan pattern hits a long tail of false positives
         (806 candidates in two_units.rsm; only ~44 land in the
         plausible PE-image VA range). The bogus ones likely come
         from procedure-LOCAL variables that share the 0x20 tag
         but encode frame offsets (not VAs) in their payload -
         the `>> 4` decode produces multi-GB junk for those.
         Filtering by .map cross-reference (next step) cleanly
         eliminates them.


================================================================================
  2026-05-22 (cont)  Step 4b: per-unit type marker decoded
================================================================================

New fixture: test/fixtures/primitives.{exe,rsm,map}. Single .dpr with
13 user globals across 12 distinct primitive types (gI / gI2 both
Integer to test same-type sharing).

[FACT]   The PRIMITIVE-form variable record has THREE sub-forms, not
         two. The previous "fixed 12-byte payload" assumption was
         wrong. New layout:

           0x20 <namelen u8> <pascal_name> 0x66 0x00 0x00
                <type_marker u8> <shifted_va u32>
                [ 0x9C 0x09 <hash u16> <trailer_type_id u16> 0xFF ]

         The 7-byte trailer in [...] is OPTIONAL. RTL globals
         (MediumBlocksLocked, LargeBlocksLocked, ...) emit the
         minimum 5-byte payload. User-code globals emit the full
         12-byte form. One edge case: the LAST user variable record
         in a unit can also use the 5-byte form (verified with gH
         in primitives.rsm).

[FACT]   The byte right after `0x66 0x00 0x00` is a PER-UNIT TYPE
         MARKER. Across all 13 globals in primitives.rsm:

           gI   Integer    -> 0x02       gSm  SmallInt -> 0x0c
           gI2  Integer    -> 0x02  ***  gL   Int64    -> 0x0e
           gC   Cardinal   -> 0x04       gU   UInt64   -> 0x10
           gB   Byte       -> 0x06       gF   Single   -> 0x12
           gW   Word       -> 0x08       gD   Double   -> 0x14
           gSh  ShortInt   -> 0x0a       gO   Boolean  -> 0x16
                                          gH   Char     -> 0x18

         Rules:
            - Variables of the same Pascal type in the same unit
              share the same marker (gI and gI2 both 0x02).
            - Each new primitive type introduced in declaration order
              gets the next even number.
            - Markers are UNIT-LOCAL: a unit where Byte is declared
              first would give Byte = 0x02.

[FACT]   The `trailer_type_id u16` in the 7-byte trailer mirrors the
         marker byte but with a +4 step (0x03ED for Integer in our
         primitives unit, 0x03F1 for Cardinal, 0x03F5 for Byte, ...).
         Same uniqueness rule: same Pascal type -> same trailer id
         within a unit; cross-unit the values shift.

[FACT]   Across-unit comparison shows the marker is stable for
         common patterns. Three different units, each with Integer
         as their FIRST declared primitive global:
            hello.S       -> marker 0x02
            two_units.S/D -> marker 0x02
            primitives.gI -> marker 0x02
         Not a coincidence; the marker is the per-unit index of the
         primitive type in declaration order, and Integer happens
         to be the first primitive declared in all three fixtures.

[FACT]   Empirically, .map next-symbol-gap is an UPPER BOUND on a
         global's size, NOT exact. Compiler padding for alignment
         shows up:
            gB (Byte, 1 byte) at 0xA75C, next gW at 0xA75E -> gap 2
            gSh (ShortInt, 1 byte) at 0xA760, next gSm at 0xA762 -> gap 2
            gO (Boolean, 1 byte) at 0xA788, next gH at 0xA78A -> gap 2
         The MIN gap across all variables sharing a marker is a
         tighter estimate (compiler doesn't typically pad between
         same-type adjacent globals).

----- Implication for step 5 (DWARF type emission) -----

Per-unit local markers + .map size gap gives us "good enough" typing
for the common case. Strategy:
   1. Group user-globals by their CompileUnit (from .map).
   2. For each unique marker in a unit, compute MIN(next-symbol-gap).
   3. Map (min_gap) -> primitive:
        1 -> Byte, 2 -> Word, 4 -> Integer, 8 -> Int64
   4. Two markers with the same MIN gap are different types but we
      can only pick one name; lowest marker wins the "popular" name.

Loses precision (Cardinal vs Integer indistinguishable; Single vs
Cardinal indistinguishable) but ships precise display for the
overwhelming majority of Delphi globals.

[QUESTION] u16-A (the hash u16 in the 7-byte trailer) increments
         by ~4 between adjacent globals in declaration order:
            gI=0x1B6D, gI2=0x1B71, gC=0x1B75, gB=0x1B79, gW=0x1B7D,
            gSh=0x1B81, gSm=0x1B85, ...
         Plausibly a sequential ID assigned by the compiler. Not
         needed for B-lite.

            0x21 <namelen u8> <pascal_name>

[FACT]   LOCAL-VARIABLE sub-record (tag 0x20):
            Same shape as parameter, just tag 0x20 instead of 0x21:
              0x20 <namelen u8> <pascal_name>
              <sub_tag triple>              (0x66 / 0x62 by type kind)
              <type_marker u8>
              <stack_offset i8>             (typically negative for locals)

         This is identical to the GLOBAL variable record shape we already
         parse, EXCEPT:
           - context: nested inside the 0x28 function record
           - location: 1-byte stack offset (vs. globals' 4-byte VA)
           - no trailer (no `0x9C 0x09 ... 0xFF`)

[FACT]   The Result pseudo-variable (return value) is encoded as a normal
         0x20 local at a fixed slot (offset 0xF8 = -8 for the two functions
         we've seen). Delphi materialises Result as a stack local and
         loads it into RAX at function exit.

[FACT]   Sub-tag triple semantic (now consistent across globals + params
         + locals):

            0x62 0x00 0x00  -- non-primitive type (record, class, enum)
         The byte AFTER the triple is the per-unit type marker (same
         numbering system as global vars).

[FACT]   Stack offsets observed for Add(a, b: Integer):
         RSM stores 1-byte signed offsets that are NOT direct byte
         offsets relative to rbp. After disassembling Geometry.Add:

           push rbp
           sub  rsp, 0x10            ; allocate 16 bytes of locals
           mov  rbp, rsp             ; rbp now points to BOTTOM of locals

           var     RSM offset   asm [rbp+...]
           ------  -----------  -------------
           A       +0x20        +0x20
           B       +0x30        +0x28        (RSM stride 0x10, asm 0x08)
           Result  -0x08        +0x0c
           Tmp     -0x10        +0x08

         The conversion formula is:
            asm_byte_offset = (rsm_offset_signed / 2) + 16
         Equivalently: the natural Delphi "frame anchor" is rbp+16
         (= the saved old-rbp slot), and RSM stores offsets in
         2-byte stride units from THAT anchor.

  2026-05-26  Real-world scale: proc-record format revisited (AdvPCB)
================================================================================

Context: M2 phase A worked on the small Geometry.Add / DistanceSq fixtures
but the original parser only recovered ~6,473 procs out of ~301,829 .map
function symbols on Altium AdvPCB (160 MB DLL, 575 MB .rsm).
Investigation via three new diagnostic subcommands:
  rsm2pdb diff-procs    -- VA-cross-check rsm procs vs .map function syms
  rsm2pdb probe-procs   -- hex dump + decode for stratified missing procs
  rsm2pdb analyze-procs -- single-pass histograms (proc subtag,
                           trailer length, sub-record subtag) over the
                           whole file

Findings revealed four independent reasons the original parser was
discarding 98% of legitimate proc records.

[FACT]   Proc-record post-name sub-tag is variable. Not always 0xA0.
         Histogram from AdvPCB (134,186 VA-confirmed proc candidates):
            0xA0  98.08%  -- normal function
            0xE0   1.46%  -- mostly RTL allocator/intrinsic helpers
                             (SysGetMem, SysAllocMem, ...)
            0x80   0.40%  -- inline / static? (e.g. WriteMemoryInt32)
            other  0.06%  -- byte-level false-positive noise
         Application rule: accept {0xA0, 0xE0, 0x80}.

[FACT]   The "head" between the name+subtag+00+00+hash(4)+VA(4) and the
         first sub-record is VARIABLE LENGTH. We had hard-coded
         `s = headEnd + 16` (i.e. trailer assumed to be exactly 5 bytes
         after VA). Reality:
            trailer len   AdvPCB share
            -----------   ------------
                  7         27.3 %
                  6         25.9 %
                  5         16.4 %   <-- the only one the old parser
                                          accepted
                  8         14.6 %
                  9          5.5 %
                  10         1.5 %
                  3..30      ~9 %    (long tail; max observed ~30)
         Application rule: scan-forward up to ~64 bytes from
         (name_end + 11) looking for EITHER a valid sub-record start
         (0x20/0x21 + 1..32 namelen + printable Pascal name + the
         double-zero invariant at the right offset) OR a real 0x63 end
         marker (one whose following byte is a recognizable record
         tag: 0x28 / 0x20 / 0x2A / 0xFF / 0x9C). A bare 0x63 inside
         the trailer is a common false positive and must be filtered
         by checking the byte after it.

[FACT]   Sub-record subtag is NOT just {0x66, 0x62}. Histogram of
         215k sub-records on AdvPCB:
            0x66  56.4%  primitive-ish, the only one we accepted
            0x16  26.0%
            0x26   8.3%
            0x62   5.0%  non-primitive-ish, also accepted
            0x12   1.9%
            0x22   1.3%
            0x06   0.6%
            tail   0.5%  (~10 more values)
         The strong invariant is the TWO zero bytes immediately AFTER
         the subtag byte; the subtag value itself is variable (almost
         certainly encodes type-class / by-ref / register-class
         attributes we don't yet decode). Application rule: accept any
         byte at subtag position; require buf[bodyAt+1] == 0 and
         buf[bodyAt+2] == 0.

         Examples of non-0x66 sub-records (from real proc records, all
         valid):
            System.@Abs:
              21 03 "Val"    16 00 00 24 20   <- param Val   subtag 0x16
              20 06 "Result" 06 00 00 24 00   <- local Result subtag 0x06
            Set8087CW:
              21 05 "NewCW"  16 00 00 10 02   <- param NewCW subtag 0x16

[FACT]   Some proc records carry sub-records with tags 0x23 / 0x25
         AFTER the param/local list. Empirically these encode return-
         value metadata and enum-entry definitions (e.g. for a function
         returning an inline enum). They DON'T fit the fixed 5-byte
         payload format. Application rule: on any tag != 0x20/0x21/0x63
         inside the walk, stop GRACEFULLY (keep the params/locals we
         already collected) instead of discarding the whole proc.

         Example: PCBCommands_PCB.TPCBCommands.FileSave (override,
         returns an inline enum):
            ... 21 04 "Self" ...
            ... 20 (var) "AInExport" ...
            ... 21 05 "AKind" 62 00 00 0a 40
            23 06 "Result" cd 02 00 00 04 50   <- 0x23: return-value descriptor
            25 06 "eASCII" 0a 00 00 e1 04 ...  <- 0x25: first enum entry
            25 08 "eBinary3" 0a ...            <- enum entry
            25 08 "eBinary4" 0a ...
            ...

[FACT]   Between two consecutive sub-records there can be a 1-2 byte
         padding / attribute prefix (probably encodes `var` / `out` /
         `const` parameter qualifiers). Without this allowance our
         scanner aligned to `bodyAt + 5` and read garbage as the next
         tag. Example, FileSave Self -> AInExport:
            21 04 "Self" 66 00 00 05 04  <- Self ends at +5 byte 0x04
            20                            <- could be local tag, but
                                             buf[+1] = 0x21 = nlen 33
                                             which fails name check
                                             (bytes 0x21 0x09 are not
                                             printable Pascal name)
            21 09 "AInExport" 66 ...     <- the REAL next sub-record
                                            starts 1 byte later
         Application rule: between sub-records, scan-forward 0..6 bytes
         for the next valid sub-record signature (same validation as
         the initial trailer scan). The 0x20 byte at offset 0 here is
         probably a `var`/`out` qualifier; we currently skip past it
         without preserving the attribute.

[FACT]   Proc-record NAMES often contain `@` (RTL helpers like
         System.@Abs, System.@DllMain, ...). Need to widen the Pascal-
         identifier set used for printable-name validation.

[QUESTION] What do the trailer bytes between VA and the first
         sub-record encode? Hypotheses (untested):
         - Return-type marker (for functions, encoded as a short
           composite). Explains "trailer of 5 bytes" being typical
           for return-typed functions but other lengths for procedures
           and/or overrides.
         - Calling-convention / linkage flags.
         - Source-line range / scope info.
         Deferred. We don't need to decode it to recover the proc.

[QUESTION] Override method's `Self` stack offset is encoded as
         RSM `off=+4` -> decoded by our formula `(off/2)+16` to
         rbp+18. That's misaligned and non-standard. For non-override
         methods (e.g. TControlsHookData.UsingStandardAtoms) Self has
         RSM off=+32 -> rbp+48, which is consistent with the Delphi
         convention discovered in M2-phase-A (args spilled starting
         at rbp+32, NOT the MS-x64 rbp+16). It's plausible that the
         off=+4 byte for override-methods' Self is a SENTINEL value
         meaning "Self lives at rbp+8 / passed in RCX, not spilled"
         rather than a literal stack offset. Either:
           - We should special-case Self in overrides (display location
             differently) and trust the rsm off only for non-Self
             params.
           - Or the RSM has a `virtual/override` discriminator we
             haven't decoded that flips the offset interpretation.
         Behavior today: Self for override methods shows up in the
         Variables panel but its displayed value is wrong. Locals
         and non-Self params are correct.

[FACT]   Pre-existing .debug_* sections in the input PE are a real
         hazard. The PE injector now RENAMES any incoming .debug_*
         sections to ".old_dw" before appending the fresh ones.
         Without this, the injected file ends up with two .debug_info
         sections, gdb reads the FIRST one (stale), and reports
         "no locals or arguments" even when our new DWARF correctly
         carries them. See src/pe/pe_injector.cpp.

[FACT] Type-table records are interleaved with 0x67 records of the
       same `<tag> <namelen> <name> <4-byte hash>` shape but
       referring to functions / methods / imports the unit uses.
       The 0x67 entries don't get a marker -- only 0x66's count
       toward the index.

       Example (04_locals "locals" unit @0x34b14e onward):

           67 07 "Writeln"        <- skip (function ref)

[FACT] Unit anchor signature (used to ground per-unit type table
       scanning): `02 <namelen> <unit-name>` followed within ~200
       bytes by `02 70 <namelen> <file>.{pas|dpr|inc|tmp}`.

       match this signature -- in 05_types only 4 of the 14 declared
       units (System, System.Character, System.SysUtils, types) are
       picked up. User-code unit is always among them, which is
       what matters for our use case (the PDB only needs accurate
       types for user-declared variables; the rest fall back to
       size-based UInt8/16/32/64).

[FACT] Sub-record stack-offset 2-byte form is SIGNED i16, NOT
       unsigned u16 as we had been decoding it. The 2-byte form is
       used when the offset doesn't fit a signed i8 (typically for
       string / bool / char locals in large frames whose true
       offsets are -200..-1600 range).

       Encoding: byte[0] has LSB=1 to discriminate from the 1-byte
       form. The two bytes form a little-endian word v; the real
       formula is:

           real_offset = sub_rsp + ((int16) v - 1) / 4

       Or equivalent normalised form (so the existing downstream
       `sub_rsp + stored_off / 2` math keeps working):

           stored_off = ((int16) v - 1) / 2

       Validated empirically on examples/05_types ProbeLocals
       string / pointer locals against the disassembled addresses
       Delphi's codegen actually uses:

           lS    v=0xFE61 (i16=-415)  -> stored=-208 -> real=448-104 = 344 = 0x158
           lAS   v=0xFE41 (i16=-447)  -> stored=-224 -> real=448-112 = 336 = 0x150
           lWS   v=0xFE21 (i16=-479)  -> stored=-240 -> real=448-120 = 328 = 0x148
           lPC   v=0xF9E1 (i16=-1567) -> stored=-784 -> real=448-392 =  56 = 0x38
           lPAC  v=0xF9C1 (i16=-1599) -> stored=-800 -> real=448-400 =  48 = 0x30
           lPWC  v=0xF9A1 (i16=-1631) -> stored=-816 -> real=448-408 =  40 = 0x28

       Earlier unsigned read of 0xFE61 as +65121 produced
       stored=+32560 -> real=448+16280 = 16728, well outside any
       real frame. All six addresses agree to the byte after the
       fix.

[FACT] Delphi-x64 prologue has two shapes (extending the
       single-block-only assumption from earlier work):

       Simple:
           55                          push rbp
           48 83 EC NN                 sub  rsp, imm8
           48 8B EC                    mov  rbp, rsp

       Multi-push (managed locals / many spills):
           55                          push rbp
           57 / 56 / 41 54..           push <reg>             zero..N
           48 83 EC NN  or  48 81 EC NN NN NN NN  sub rsp, imm
           48 8B EC                    mov  rbp, rsp

       Each `push <reg>` between `push rbp` and `sub rsp` shifts
       the saved-rbp / return-addr / rcx-shadow slot up by 8.
       Locals (rbp+0 .. rbp+sub_rsp) are not affected.

       Formula recap:
           real_local = sub_rsp + RSM/2
           real_param = sub_rsp + 8*extra_pushes + RSM/2
           real_self  = sub_rsp + 8*extra_pushes + 16

[FACT] Nested-function proc-record subtag is 0x41 (additional to the
       known 0xA0 / 0xE0 / 0x80 / 0x20 set). Format differs:

           0x28 <namelen> "<name>" 0x41 02 10 00 <VA u32> ...

       There are 3 mystery bytes (`02 10 00`) between the subtag
       and the VA u32 (vs `00 00 <hash u32>` in the 0x80/A0/E0
       forms). The implicit static-link parameter (parent's rbp,
       passed in rcx) is NOT recorded as a sub-record; the parser
       sets `has_static_link = true` when it sees subtag 0x41 so
       downstream can synthesise a `__frame_outer__` local.

[FACT] Self marker palette (all should bypass the general
       sub_rsp+RSM/2 decode and use sub_rsp + 16 + 8*extra_pushes
       directly):

           0x29  base / virtual / regular class method Self
           0x31  override-method Self  (paired with sentinel +4109)
           0xD5  anonymous-method closure Self  (paired with +26)

       The `name == "Self"` fallback also catches all three so we
       don't have to enumerate every marker variant.

[FACT] String type display in CodeView is achieved via
       SimpleTypeKind + SimpleTypeMode::NearPointer64 (no
       LF_POINTER record needed):

           UnicodeString / WideString / "string" / PChar / PWideChar
               -> TypeIndex(WideCharacter, NearPointer64)
                  = T_64PWCHAR = 0x0671
                  cdb displays: wchar_t * "..."

           AnsiString / UTF8String / PAnsiChar
               -> TypeIndex(NarrowCharacter, NearPointer64)
                  = T_64PRCHAR = 0x0670
                  cdb displays: char * "..."

       Pascal RTL conveniently puts char data at the pointer's
       exact address (the RTL header bytes live just below 0), so
       cdb's "pointer-to-char string display" mode just works

       ShortString is NOT a managed pointer -- it's an inline

       composite hash-ref + 1-byte offset:
         00 <vis 1B> 00 <type_hash u16 LE> <offset_x2 u8>
         <internals 8B> <parent_hash u16 LE> ff
         total: 17 bytes after name

       composite hash-ref + 2-byte offset:
         00 <vis 1B> 00 <type_hash u16 LE> <offset_low u8>=odd <offset_high u8>
         <internals 8B> <parent_hash u16 LE> ff
         total: 18 bytes after name

       Parser robustness: don't rely on total length -- walk-to-ff
       is the canonical stop. Internals can carry varying back-refs.

[FACT] Disambiguating type-ref form (1-byte primitive marker vs
       2-byte composite hash). Empirically:

         - Primitive markers are small even bytes from the table
           kPrimitiveTable (currently 36 entries; max marker
           ~0x42). All bytes < 0x60.
         - Composite hashes seen in records.rsm range 0x1a6d..
           0x1aa1 in low byte (so low byte > 0x40 in this fixture),
           with 0x1a or 0x1b as high byte. AdvPCB has hashes
           starting at 0x29 (low) -- overlap with the primitive
           marker range IS possible in principle.

       Safe parser rule: at the type-ref position, attempt to
       resolve the byte as an `own_hash` against the unit's
       collected type-record map (and verify the NEXT byte is
       consistent with the hash's expected high-byte). If yes,
       it's a composite ref. If no, treat as 1-byte primitive
       marker (and validate against kPrimitiveTable).

[FACT] AdvPCB.rsm size: 576,706,973 bytes. Format-wise indistinguish-
       able from our small records.rsm fixture -- same tag bytes,
       same record shapes, same self-describing offset/type-ref
       forms. Confidence for Step 11b implementation: HIGH.

Step 11b is now unblocked. Phase B (parser extension) can proceed
without speculative branches.

2026-05-28 (later) -- class-header layout is variable-length
---------------------------------------------------------------------------

Phase A's first pass claimed the class-header intermediate block
(between `47 00 10 00 00 <own_hash u16>` and the base-class slot)
was a fixed 11-byte "vtable size info". That held for
examples/07_records (TShape / TCircle / TBag all have 2 declared
methods each: Create + Describe / Create + Describe + Create+Free).
Broke immediately on examples/08_inherit_props, where TDog declares
the additional property accessors GetBarkCount + SetBarkCount and
the block grew by 1 byte.

[FACT] Empirical sizes seen so far:

         07_records TShape   (Create + Describe)      9-byte block
         07_records TCircle  (Create + Describe)      9-byte block
         07_records TBag     (3 fields, no methods)   9-byte block
         08 TAnimal          (Create + Describe)      10-byte block
         08 TMammal          (Create + Describe)      10-byte block
         08 TDog             (Create + Describe +
                              GetBarkCount + SetBarkCount)
                                                      10-byte block

       Sample TMammal class header (from inherit_props.rsm):

         47 00 10 00 00          -- header marker
         79 1a                   -- own_hash (TMammal)
         99 07                   -- u16 (vtable count? always small)
         46 dd 01                -- 3 bytes (unique to fixture)
         10 cd 1b                -- another 3-byte block
         00 00                   -- separator
         33 83                   -- end marker of vtable block
         71 1a                   -- base_hash (TAnimal)
         00 40                   -- start of 8-byte tail
         c9 1b 02 fe 00 04       -- ... rest of tail
         25 3f 00                -- field-list anchor

       Equivalent TCircle (07_records) is one byte shorter inside
       the `46 ... 33 83` window. Not yet clear which dimension
       drives the growth -- possibly bytecount-of-vtable, possibly
       number of distinct method signatures the linker emitted.
       For now we don't decode this block; we just walk past it.

[FACT] Robust base_hash location. The ONLY invariant across
       fixtures is the 8-byte "tail" between the base-class slot
       and the field-list anchor:

         <base_hash u16> <00 NN .. fe 00 NN> <25 3f 00>

       So: scan forward from `47 00 10 00 00 <own_hash>` (up to
       ~80 bytes) for `25 3f 00`; base_hash is at fl_at - 10 .. fl_at - 9.

       Implementation lives in src/rsm/rsm_reader.cpp scanAggregateTypes
       (since commit 394e12a). Verified on:
         07_records: TCircle.base_hash = TShape ✓ (no change)
         08_inherit_props:
             TAnimal.base_hash = 0x0002 (no local match -> no base)
             TMammal.base_hash = TAnimal ✓
             TDog.base_hash    = TMammal ✓

       Side-effect: the old fixed-offset version sometimes
       picked up adjacent RTL hash bytes as a spurious base_hash,
       which pulled extra RTL aggregates into the PDB TPI
       (REGSAM, etc.). Those are gone now.

[FACT] Pascal `property` declarations are not represented in RSM
       at all. The compiler emits getter / setter procedures as
       normal methods (regular 0x2e records with the property
       accessor names like GetBarkCount / SetBarkCount), and the
       backing fields as 0x2c records under the class. The
       property NAME is purely a source-language artifact and
       has no debug-info counterpart. cdb / VS surface the
       methods + fields but not the property -- documented as
       gotcha #28 (docs/06-gotchas.md).

2026-05-28 (night) -- cross-unit type-reference encoding
---------------------------------------------------------------------------

Discovered via examples/09_cross_unit (Shapes / Items / Layout /
cross_unit.dpr layered fixture). Three groups of empirical
facts; collectively they let cross-unit composite fields
(Items.TItem.fPos -> Shapes.TPoint with X / Y fields) round-trip
into the PDB.

[FACT] Unit anchor source-file marker has TWO byte patterns.
       The `02 <nl> <unit_name>` anchor is followed within ~200
       bytes by a source-file signature. Empirically:

         02 70 <fnl> <filename>     -- used by .dpr-style
                                       self-anchors (the .dpr
                                       finds its own source via
                                       this).
         3c 70 <fnl> <filename>     -- used by .pas unit anchors
                                       (Shapes / Items / Layout
                                       in 09_cross_unit; AdvPCB
                                       RTL units routinely).

       The trailing structure (`fnl` byte, filename, optional
       `.pas` / `.dpr` / `.inc` / `.tmp` extension) is the same
       in both cases. Parser accepts either prefix.

[FACT] Per-unit 0x66 type table = cross-unit IMPORT TABLE.
       Each 0x66 entry has shape:
           66 <nl> <name> <4-byte hash trailer>
       For each Pascal type the unit REFERENCES (declared OR
       imported), the unit's 0x66 table carries an entry with
       that type's globally-unique 4-byte hash.

       Examples from 09_cross_unit Items.pas region (0x34b62e):
         66 07 TObject              4byte=0x93630ec3
         66 0d UnicodeString        4byte=0xcb0cbc2c
         66 07 Integer              4byte=0xb573cea0
         66 06 string               4byte=0xcb0cbc2c
         66 06 TPoint               4byte=0x0c6d2cf6  <- imported
         66 05 TSize                4byte=0x3ed57bdf  <- from
         66 06 TColor               4byte=0x5e4acac6  <- Shapes

       cross_unit.dpr's 0x66 table contains the same hash4 for
       TPoint (0x0c6d2cf6) plus its own imports (TItem
       0x95e4424c, TLayout 0x14e07925).

       The LOW 16 BITS of a 4-byte hash equal the type's
       own_hash in its declaring unit. For TPoint (Shapes.pas
       declares it with own_hash 0x2cf6), low 16 of 0x0c6d2cf6
       is 0x2cf6 -- exact match.

       Cross-unit resolution algorithm:
         field.primitive_marker
              -> unit's 0x66 entry at position marker/2 - 1
              -> (name, hash4)
              -> if name is primitive, resolve via
                 kPrimitiveTable.
              -> else search aggregates() for one with
                 own_hash == (hash4 & 0xFFFF) AND name match.
                 That's the declaring-unit aggregate.

[FACT] The 0x66 table CAN have multi-uses SEPARATORS between
       blocks. When a unit `uses` multiple other units, blocks
       of 0x66 entries are interleaved with short separator
       records:
           63 64 <fnl> <imported_unit_name> 00 00 00
       (length 6 + fnl bytes). Empirically Items.pas has
           63 64 07 SysInit 00 00 00
           63 64 06 Shapes  00 00 00
       between its TObject-method 0x67 entries and the
       Shapes-imported 0x66 entries (TPoint / TSize / TColor).

       Parser must walk past these and continue collecting
       0x66/0x67 records, otherwise cross-unit imports get
       lost (the loop stops at the first non-(0x66/0x67) byte).

[FACT] Same-unit "4 hashes per class" pattern.

       A Pascal class declared within ONE unit carries FOUR
       distinct 2-byte hashes serving four different roles:

         NAME       = 0x2a own_hash
         LINKED     = 0x2a's secondary hash @ +7..+8 from kind
                      (only present when kind has bit 0x80 set)
         CLS_HEADER = 0x47 own_hash
         FLD_PARENT = 0x2c parent_hash (what fields parent to)

       Samples:
         records.dpr TPoint        : own=0x1a71 (old format,
                                     no linked_hash; direct
                                     attach works)
         09 Items.TItem            : NAME=0x424c, LINKED=0x0712,
                                     CLS_HEADER=0xb112,
                                     FLD_PARENT=0x1208
         09 Layout.TLayout         : NAME=0x7925, LINKED=0x0710,
                                     CLS_HEADER=0x7d10,
                                     FLD_PARENT=0x1008
         09 Shapes.TPoint (record) : NAME=0x2cf6, LINKED=0x0706,
                                     no CLS_HEADER (records
                                     have no 0x47),
                                     FLD_PARENT=0x0608
         09 Shapes.TSize  (record) : NAME=0xe97c, LINKED=0x8708,
                                     FLD_PARENT=0x0808
         AdvPCB TPCBCommands       : NAME=0xd0cf, LINKED=0x0405,
                                     CLS_HEADER=0x0405,
                                     FLD_PARENT=0x0405
                                     (all three equal --
                                     "direct match" form)

       Empirical formulas (universal across records + classes):
         FLD_PARENT = ((LINKED & 0xff) << 8) | 0x08

       For CLASSES specifically two patterns observed:
         pattern A (AdvPCB-style):  LINKED == CLS_HEADER own_hash
         pattern B (09-style):      LINKED.low == CLS_HEADER.low
                                    (high bytes differ)

       Resolution (src/rsm/rsm_reader.cpp, "Phase X.3"):
         1. Compute FLD_PARENT from LINKED via formula above.
            Register aggregate under FLD_PARENT hash in
            aggr_by_unit_hash_ so fields find it.
         2. If class_hashes already has LINKED -> AdvPCB direct
            match, kind = Class.
         3. Else search class_headers for one with own_hash low
            byte matching LINKED.low (09 pattern). If unique
            match, register under CLS_HEADER own_hash too;
            kind = Class.
         4. Classifier respects already-set kind to avoid
            re-classifying chained Class as Unknown.

[GUESS] AdvPCB and 09 use the same `kind=0xa8` (0x80 bit set)
       but the LINKED-vs-CLS_HEADER relationship differs.
       Probably reflects compiler-version differences in the
       per-class linker encoding. Both patterns covered by the
       resolution logic above.

[FACT] 09-format enum entries use a DIFFERENT marker layout
       than AdvPCB. AdvPCB uses
           25 <nl> <name> 0a 00 00 <parent u16> 00 00 <ord_x2>
       and parent_hash sits at +3..+4 from the `0a`.
       09 uses
           25 <nl> <name> 8a 00 00 <entry_id u32> 04 00 00 <ord>
       with NO direct parent_hash field at that offset. Each
       entry carries its own sequential 4-byte ID; the repeated
       value across all entries (`48 f1 04` at +5..+7 for the
       clRed..clYellow batch) might be a back-pointer but
       doesn't match TColor's own_hash directly.

       Consequence: TColor cross-unit field still renders as
       `unsigned` in 09 -- the parent linkage hasn't been
       decoded yet. Records (TPoint, TSize) work; enums don't.
       This is a known limitation called out in CLAUDE.md.

2026-05-29 (Tier 2 / NatVis) -- Pascal `property` records (tag 0x31)
---------------------------------------------------------------------------

CORRECTION to 2026-05-28 [FACT] above ("Pascal `property`
declarations are not represented in RSM at all"). They ARE
represented -- with a dedicated record tag 0x31. The old
conclusion came from looking only for property records right
next to their backing fields; the actual property records live
in a DIFFERENT cluster of the RSM metadata stream, further
along from the 0x2c/0x2e cluster.

[FACT] Property record format (tag 0x31). Located by scanning
    for `0x31 <namelen u8> <printable name>` followed by the
    `00 02 00 <type> fe 0f 00 00 00 80` signature.

    Layout (variable length body):

      31 <namelen u8> <PropName>
      00 02 00 <type_marker u8>      ; type kind / index
      fe 0f 00 00 00 80              ; fixed signature
      <getter_ref>                   ; W bytes (W = 1 or 2; see below)
      <setter_ref>                   ; W bytes; all-zero = read-only
      <tail>                         ; variable; ends before next record

    Followed by the usual per-record trailer
      9c <flag> <self_marker> 07 00 00 08 <2 bytes> ff

    The trailer's `9c <flag>` byte selects 1-byte (`9c 01`) vs
    2-byte (`9c 09`) marker width for the WHOLE record family
    (fields + methods + properties inside the same unit).
    Mixed widths within one RSM file haven't been observed.

[FACT] Accessor marker width is per-RSM-version (or per-Delphi-
    version):

    examples/09_cross_unit (1-byte markers, `9c 01 <byte>`):
      Tag        (RW field-backed, Integer) body = 90 96 96 00 00 00 0f 00 00 00 80
      Tag2       (mixed: f-read, m-write)    body = 06 96 72 00 00 00 0f 00 00 00 80
      Name       (RO field-backed, string)   body = 04 8e 00 00 00 00 00

    examples/08_inherit_props (2-byte markers, `9c 09 <u16>`):
      BarkCount  (RW method-backed, Integer) body = 06 45 1c 51 1c 00 00 00 0f 00 00 00 80
      Name       (RW field-backed, string)   body = 04 dd 1c dd 1c 00 00 00 00 9c 01 ed 1c
      Breed      (RO field-backed, string)   body = 04 05 1d 00 00 00 00 00 00

    The accessor markers point at per-class field/method
    counters, NOT at type-table hashes. Each 0x2c (field) and
    0x2e (method) record in the class carries its own marker
    byte/word right after its `9c 01`/`9c 09` sub-record header:
      9c 01 <field_marker_u8>     (for fields, 09 fixture)
      9c 09 <field_marker_u16>    (for fields, 08 fixture)
      e2    <method_marker_uW>    (for methods; the `e2` prefix
                                  replaces `9c 0?` for methods.
                                  W follows the same rule.)
    Property's getter / setter slot stores the marker value
    found in the corresponding field's or method's record.

    Same-class field markers increment by 2 in declaration
    order. Methods increment differently (param-count-dependent
    step?). RO properties have setter_ref = 0.

[FACT] Property records live in a SEPARATE metadata cluster from
    the field/method cluster of the same class.

    In 09_cross_unit, TItem class has:
      Field cluster   @0x34b62b (5 fields + 2 methods, no
                                 property records nearby).
      Property cluster @0x34d8f9 (7 property records back-to-
                                 back).

    Same cluster split observed for TLayout (Layout.pas) and
    in 08_inherit_props for TAnimal / TMammal / TDog. Roughly
    +0x2000 bytes downstream from the field cluster, no
    universal alignment.

    To link properties back to their owning class, the parser
    needs the property's PARENT_HASH. This sits inside the
    property's `9c 01 ... 07 00 00 08 <2 bytes> ff` trailer:
    the `<2 bytes>` immediately before `ff` is the class's
    own_hash (matches 0x47 CLS_HEADER for classes).

    Verified: TItem properties' trailer 2-byte parent is
    `14 ff` -> 0x1408 + the LINKED-hash-low-byte pattern.
    Cross-check on Pos's trailer: `9c 01 a0 07 00 00 08 14 ff`
    -> last 2 bytes before ff = `08 14` -> low-byte 0x14 ==
    TItem's NAME-hash low byte 0x4c. NOT exact match -- this
    one needs more RE before relying on it; for Tier 2.0 the
    parser instead resolves properties by NAME-PROXIMITY to
    the most recently parsed class (cluster heuristic).

[DECISION] Tier 2.0 implementation strategy.

    The full accessor-marker resolution (look up getter_ref in
    a per-class field+method marker table, fall back across
    1-byte / 2-byte widths) is non-trivial and the parent-hash
    encoding still has open RE questions. Going with a NAMING
    HEURISTIC instead for the first cut:

      Given property NAME P inside class C:
        a. If C has field "fP"          -> emit `<Item Name="P">fP</Item>`
        b. Else if C has method "GetP"  -> emit `<Item Name="P">GetP()</Item>`
        c. Else                          -> skip (rare)

    Covers ~95% of real-world Delphi (where the `f` /
    `Get` prefix is the dominant convention). Edge cases like
    09's `Tag2` (reads `fTag` not `fTag2`) get skipped.

    Tier 2.1 (deferred): proper accessor-marker resolution.
    Reactivation criteria: a real Delphi codebase showing
    properties with non-conventional accessor names that
    users actually want to inspect via Watch.

2026-05-29 (Tier 2 / cross-unit local typing) -- PARKED
---------------------------------------------------------------------------

While verifying Tier 2 (Pascal property NatVis emission) on the
extended 09_cross_unit fixture, surfaced that cross-unit class-as-
LOCAL type resolution was never working end-to-end. cdb / VS Watch
shows:

    lItem    : struct TDisposeRec       (should be TItem)
    lLayout  : unsigned __int64         (should be TLayout)

[FACT] Pre-existing since Phase B.2 (commit 3119007), confirmed by
    re-testing the original 09_cross_unit fixture at commit
    c787057 (pre-session start). Not a regression introduced by
    Tier 2 work.

[FACT] Root cause -- in Reader::resolveAggregateTypes (rsm_aggregates
    .cpp resolveAggrFor):

      a = findAggregateInUnit(ua, v.inline_type_id);
      if (!a) a = findAggregateByHash(v.inline_type_id);

    For SAME-unit class locals (08_inherit_props.lDog), inline_type_id
    equals the aggregate's own_hash in its declaring unit, so
    findAggregateInUnit hits and pascal_type stamps correctly
    ("TDog").

    For CROSS-unit class locals (09_cross_unit.lItem), inline_type_id
    is a different encoding -- a per-procedure trailer-id from a
    GLOBAL type-id table at file offset ~3516606..3522890 (cross_unit
    .rsm), NOT the foreign aggregate's own_hash. findAggregateInUnit
    misses (TItem not in cross_unit's anchor), and findAggregateByHash
    falls back to global last-wins which hits unrelated RTL types
    (TDisposeRec / IDisposer / etc.) that happen to share the
    trailer-id value as their own_hash.

[FACT] Trailer-table structure (cross_unit.rsm):

      offset 3516606..3522890     -- contiguous global type-id table
      1571 entries, each 4 bytes  -- `<id u16 LE> <kind u8> 0x0f`
      kind byte mostly 0x10       -- (occasionally 0x20 / 0x30)
      ids mostly +4 sequential    -- with occasional jumps between
                                     hash families (0x06XX -> 0x1bXX)

    cross_unit's 5 locals (lPoint .. lLayout) occupy entries
    248..255 (ids 0x0641 .. 0x065d).

[GUESS] The trailer table is a global INDEX into a parallel data
    structure that maps each id to (unit, name, hash). That parallel
    structure has not yet been identified -- 0x66 import-table entries
    don't carry the matching id, and named search around the trailer
    table boundaries didn't reveal a names array.

[DECISION] PARKED. Not blocking Tier 2 property emission (which
    works correctly on single-unit fixtures like 08_inherit_props
    where this bug isn't triggered). Cross-unit local-typing is
    its own RE round -- the same class as the 8a-enum-linkage gap
    parked on 2026-05-29-day. NatVis sidecar for cross_unit
    classes is emitted (1138 bytes, 11 properties), but VS can't
    apply our `<Type Name="TItem">` rule because lItem in PDB
    is typed as TDisposeRec (whatever findAggregateByHash returned),
    not TItem.

    Reactivation criteria: a user-reported request for cross-unit
    class-as-local inspection on a real Delphi project where this
    pattern matters. The next RE round needs to find the inline_type_id
    -> (name, hash) map; candidate places to look:
      - Cross-reference each global trailer-table entry against the
        consuming unit's 0x66 import table for sequential pairing.
      - Look for additional named-type tables near the global
        trailer table boundaries.
      - Check if the variable record's flag byte after inline_type_id
        (0xf0/0xe0/... for cross_unit locals) encodes a unit ID.

[FACT] inline_type_id of a composite local is the local's SLOT-ID,
    read as the 2-byte value at the sub-record's marker position:
        20 <len> <name> 66 00 00 <slot_id u16 LE> <flag u8>
    ProbeAll's locals (post-Extra build):
        lPoint   slot=0x064d   flag=0xf0
        lSize    slot=0x0651   flag=0xe0
        lColor   slot=0x0655   flag=0xde
        lItem    slot=0x0659   flag=0xc0
        lLayout  slot=0x065d   flag=0xb0
        lExtra   slot=0x0661   flag=0xa0
        lExtra2  slot=0x0661   flag=0x90      <-- SAME slot as lExtra

[FACT] The slot-id is PER-TYPE-PER-PROCEDURE, not per-occurrence.
    lExtra and lExtra2 are both TExtra and share slot 0x0661; they
    differ ONLY in the trailing flag byte (0xa0 vs 0x90). So the
    flag byte is the per-occurrence / stack discriminator (it
    decreases per local), and the slot-id identifies the TYPE.
    This is THE key fact the earlier parks missed -- they assumed
    one id per variable.

[FACT] The slot-id linearly indexes the unit's own 0x66 IMPORT
    BLOCK (the contiguous run of `66 <len> <name> <hash4 u32 LE>`
    entries, interleaved with 0x67 method refs, that lists every
    type the unit can see -- RTL + its `uses` types):

        slot_id = base_unit + 4 * position_in_import_block

    where position_in_import_block = the entry's index counted over
    ALL 0x66 entries from the file start (the global 0x66 index,
    "gidx"), and base_unit is a per-unit constant.

    Verified across the WHOLE cross_unit import block (gidx
    727..1293, 567 entries), C = base - 4*gidx_start = -1887:
        gidx 727  {System}TArray<...TLangRec>  slot 0x03fd  (base)
        gidx 875  TPoint                        slot 0x064d
        gidx 876  TSize                         slot 0x0651
        gidx 877  TColor                        slot 0x0655
        gidx 878  TItem                         slot 0x0659
        gidx 879  TLayout                       slot 0x065d
        gidx 880  TExtra                        slot 0x0661   <-- new
    The 6 user types sit CONSECUTIVELY at the tail of the block
    (gidx 875..880), so their slots step +4 in declaration order.
    Adding TExtra appended exactly one entry (gidx 880 -> 0x0661)
    without shifting the others.

[FACT] The bridge therefore is:
        local.slot_id
          -> position = (slot_id - base_unit) / 4
          -> unit_import_block[position] -> (name, hash4)
          -> findAggregateByHash(hash4) / hash4-low16 == own_hash
          -> AggregateType (TItem / TLayout / TExtra / ...)
    hash4 -> aggregate already works (gotcha #35: hash4 is the
    universal per-type id, low 16 bits == own_hash in declaring
    unit). The missing piece was ONLY slot_id -> hash4.

[DISPROVEN] base_unit is NOT global (no single C for the file).
    TPoint appears in the 0x66 table at gidx {136,460,468,875,902}
    (once per importing unit). cross_unit uses gidx 875 (C=-1887).
    A global C would give Items' TPoint (gidx 460) slot
    4*460-1887 = -47 (impossible). So each unit numbers its own
    import block; base_unit differs per unit.

[FACT] Master slot table (the OLD "trailer table"), now @3522817:
    contiguous 4-byte entries `<slot_id u16 LE> <kind u8> 0x0f`,
    slot-ids ASCENDING +4 (1561/1578 steps are exactly +4; 17
    jumps). kinds: 0x10 x1567, 0x20 x10, 0x30 x2. This is the
    global, slot-sorted master list of ALL slot-ids in the program.
    It carries NO hash4/name, so it cannot map slot->type on its
    own. It is NOT 1:1 with the 0x66 table by position
    (master[259]=0x0661 but 0x66[259]=Pointer; the two tables are
    different orderings -- master is slot-sorted, 0x66 is gidx /
    unit-declaration order).

[DISPROVEN] base_unit is NOT recoverable from the master table's
    structure: 0x03fd (cross_unit's base) sits at master pos 128 in
    the middle of a clean +4 run (...3f5,3f9,3fd,401,405...); the
    kind!=0x10 markers and the +4 jumps do NOT line up with unit
    boundaries. base_unit is also NOT stored adjacent to the import
    block start (no `fd 03` near @3472017). The linker almost
    certainly computes base_unit as a cumulative type-count carried
    in the .dcu, which RSM does not expose directly.

[METHOD] Proposed robust implementation -- self-calibrate base_unit
    per unit, no magic constant, no .dcu needed:
      1. Parse each unit's 0x66 import block in order -> list of
         (name, hash4, gidx). Flag entries whose hash4 resolves to
         a known AggregateType.
      2. Collect the slot-ids referenced by that unit's cross-unit
         (unresolved) locals/params.
      3. Solve for the single base_unit such that EVERY referenced
         slot maps (pos=(slot-base)/4) onto an aggregate entry.
         Aggregates are sparse in the import block, so the
         alignment is over-determined and unique.
      4. Optional cross-check: compose's sniffed byte_size for the
         local (class -> 8-byte pointer, record -> record size)
         must agree with the resolved aggregate's kind/size.
    With cross_unit's 6 locals the alignment is heavily
    over-determined -> robust. Implementation site:
    Reader::resolveAggregateTypes (rsm_aggregates.cpp resolveAggrFor),
    which currently does findAggregateInUnit(ua,inline_type_id)
    then findAggregateByHash(inline_type_id) -- both wrong for the
    cross-unit slot-id case (the latter hits RTL collisions like
    TDisposeRec).
